```typescript
if (!res.ok) {
```


This gives us `error` (for rendering ErrorEmptyView) and `revalidate` (for recovery actions). The retry wrapper handles 3 retries with 1s/2s/4s backoff. During retries, `isLoading` remains true -- EmptyView won't show until retries are exhausted (per Raycast behavior: EmptyView hidden when isLoading=true and search bar empty).


Wrap the entire try/catch to catch both network errors (TypeError: fetch failed) and HTTP errors (thrown Error).

If the fetch fails, the user is already looking at Firefox (or their desktop). showActionError will show a toast which Raycast still displays even when the window is closed. This is acceptable behavior -- the toast floats above other windows.

**2. Update closeTab:**
Replace the current catch block:
```typescript

  toast.title = "Failed to close tab";
  toast.message = "Could not connect to Firefox";
}
```
With:
```typescript
} catch (error) {
  toast.hide();
  await showActionError(error, "close tab");
}
```
Hide the animated toast first, then show the classified failure toast. This prevents two toasts stacking.

Also update the `mutate()` call -- the fetch inside mutate needs to throw on !res.ok so errors propagate:
```typescript
await mutate(
  (async () => {
    const res = await fetch(`http://127.0.0.1:${port}/close`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ tabId }),

    if (!res.ok) {
      const body = await res.json().catch(() => ({ error: "Unknown error" }));

    }
  })(),
  { optimisticUpdate(data) { return (data ?? []).filter((t) => t.id !== tabId); } },
);
```

Per user decision: keep tab list visible on action failure (toast only), no list disruption.
