## Summary

This phase adds two GitHub Actions workflows to automate CI checks and release publishing. The project currently has three components (`extension/`, `native-host/`, `raycast-extension/`) each with their own `package.json` but NO root-level `package.json` and NO workspace configuration. Since Changesets requires npm workspaces to discover packages in a monorepo, a root `package.json` with workspace declarations must be created as a prerequisite.

A critical finding is that the `changesets/action` does NOT create GitHub Releases for private packages -- it only creates git tags when `privatePackages.tag` is `true`. Since all three packages are `"private": true`, the release workflow must use a separate action (`softprops/action-gh-release@v2`) to create the actual GitHub Release with assets after changesets creates the tag.

The recommended architecture uses two workflows: (1) a CI workflow on PRs and main pushes that builds/lints/tests all components, and (2) a release workflow that runs changesets version/tag on main pushes, then creates a GitHub Release with native host assets when a `native-host@*` tag is detected. The GITHUB_TOKEN is safe to use because pushes made with it do NOT trigger subsequent workflows, naturally preventing infinite loops.

**Primary recommendation:** Use npm workspaces + Changesets for versioning/tagging, with `softprops/action-gh-release@v2` handling the actual GitHub Release creation and asset upload for native-host tags.

### Pattern 1: Root package.json with npm Workspaces

**What:** A root `package.json` that declares all three component directories as workspaces. Required for Changesets to discover packages.

**Why needed:** Changesets ONLY works with single-package repos or workspace-based monorepos. Without workspace config, `changeset init` will not detect sub-packages.

**Example:**
```json
{
  "name": "raycast-firefox",
  "version": "0.0.0",
  "private": true,
  "workspaces": [
    "extension",
    "native-host",
    "raycast-extension"
  ],
  "devDependencies": {
    "@changesets/cli": "^2.29",
    "@changesets/changelog-github": "^0.5"
  },
  "scripts": {
    "version": "changeset version",
    "release": "changeset publish"
  }
}
```

### Pattern 2: Changesets Config for Private Packages

**What:** Configure changesets to version and tag private packages (since none of these publish to npm).

**Example:** `.changeset/config.json`
```json
{
  "$schema": "https://unpkg.com/@changesets/config@3.1.1/schema.json",
  "changelog": ["@changesets/changelog-github", { "repo": "user/raycast-firefox" }],
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": [],
  "privatePackages": {
    "version": true,
    "tag": true
  }
}
```

**Critical:** `privatePackages.tag: true` is required so `changeset publish` creates git tags like `native-host@1.0.0`. Without this, no tags are created and the release workflow never fires.

### Pattern 3: CI Workflow (PRs + main pushes)

**What:** Validates all three components on every PR and main push.

**Example:** `.github/workflows/ci.yml`
```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci

      # Native host
      - name: Build native host bundle
        run: npm run build --workspace=native-host
      - name: Verify bundle exists
        run: test -f native-host/dist/host.bundle.js

      # Extension lint
      - name: Lint Firefox extension
        run: npm run lint --workspace=extension

      # Raycast extension
      - name: Lint Raycast extension
        run: npm run lint --workspace=raycast-extension
      - name: Build Raycast extension
        run: npm run build --workspace=raycast-extension
```

### Pattern 4: Release Workflow with Changesets + softprops/action-gh-release

**What:** On main push, changesets creates a version PR. When that PR merges (consuming changesets), the action runs `changeset publish` which creates tags. A subsequent step detects `native-host@*` tags and creates a GitHub Release with assets.

**Key insight:** Because the changesets action uses GITHUB_TOKEN, its version-bump commits do NOT trigger new workflow runs (preventing infinite loops). The release job must happen in the SAME workflow run.

**Example:** `.github/workflows/release.yml`
```yaml
name: Release

on:
  push:
    branches: [main]

concurrency: ${{ github.workflow }}-${{ github.ref }}

jobs:
  release:
    runs-on: macos-latest
    permissions:
      contents: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci

      - name: Build native host
        run: npm run build --workspace=native-host

      - name: Create Release Pull Request or Publish
        id: changesets
        uses: changesets/action@v1
        with:
          publish: npx changeset publish
          version: npx changeset version
          commit: "chore: version packages"
          title: "chore: version packages"
          createGithubReleases: false  # We handle releases ourselves
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: Generate SHA256 checksum
        if: steps.changesets.outputs.published == 'true'
        run: |
          cd native-host/dist
          shasum -a 256 host.bundle.js run.sh > SHA256SUMS.txt
          # Round-trip verify
          shasum -a 256 -c SHA256SUMS.txt

      - name: Get native-host version
        if: steps.changesets.outputs.published == 'true'
        id: nh-version
        run: |
          VERSION=$(node -p "require('./native-host/package.json').version")
          echo "version=$VERSION" >> "$GITHUB_OUTPUT"
          echo "tag=native-host@$VERSION" >> "$GITHUB_OUTPUT"

      - name: Create GitHub Release
        if: steps.changesets.outputs.published == 'true'
        uses: softprops/action-gh-release@v2
        with:
          tag_name: ${{ steps.nh-version.outputs.tag }}
          name: native-host@${{ steps.nh-version.outputs.version }}
          generate_release_notes: false
          files: |
            native-host/dist/host.bundle.js
            native-host/dist/run.sh
            native-host/dist/SHA256SUMS.txt
```

### Pattern 5: SHA256 Round-Trip Verification

**What:** Generate checksum, then immediately verify it, ensuring the file on disk matches what was recorded.

**Example:**
```bash
cd native-host/dist
shasum -a 256 host.bundle.js run.sh > SHA256SUMS.txt
# Round-trip: verify the checksums we just generated
shasum -a 256 -c SHA256SUMS.txt
```

**Note:** macOS provides `shasum` (not `sha256sum`). The `-a 256` flag selects SHA-256. The output format is compatible with `sha256sum` on Linux.

### Anti-Patterns to Avoid
- **Using a PAT for changesets/action:** Causes infinite workflow triggers when changesets pushes version commits. GITHUB_TOKEN prevents this by design.
- **Triggering release on tag push separately:** Since GITHUB_TOKEN pushes don't trigger workflows, a separate `on: push: tags:` workflow will NEVER fire from changesets-created tags. The release must happen in the same workflow run.
- **Using `changesets/action` `createGithubReleases: true` with private packages:** This will silently produce no releases because private packages are not "published" in changesets' view.
- **Forgetting `privatePackages.tag: true`:** Without this, `changeset publish` for private packages creates no tags and no releases.
- **Running `npm install` instead of `npm ci`:** `npm install` updates lockfile; `npm ci` does a clean install from lockfile. Always use `npm ci` in CI.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Version management | Custom version bump scripts | Changesets CLI | Handles changelogs, tag naming, monorepo coordination |
| Changelog generation | Manual CHANGELOG.md editing | `@changesets/changelog-github` | Automatic PR links, author attribution, consistent format |
| Release asset upload | `gh release create` in raw bash | `softprops/action-gh-release@v2` | Handles idempotent release creation, glob patterns, error handling |
| npm dependency caching | Manual `actions/cache` for node_modules | `actions/setup-node` with `cache: npm` | Built-in, handles cache key generation automatically |
| Git user config in CI | Manual `git config` commands | `changesets/action` `setupGitUser: true` (default) | Configures `github-actions[bot]` identity automatically |

**Key insight:** The changesets ecosystem handles the complex orchestration of version bumps, changelogs, and tagging across a monorepo. The gap is only GitHub Release creation for private packages, which `softprops/action-gh-release` fills cleanly.

## Common Pitfalls

### Pitfall 1: No Root package.json / No Workspaces
**What goes wrong:** `changeset init` or `changeset add` fails to detect sub-packages, treating the repo as a single-package project.
**Why it happens:** Changesets discovers packages through npm/pnpm/yarn workspace declarations. Without a root `package.json` with `"workspaces"`, sub-directories are invisible.
**How to avoid:** Create a root `package.json` with `"private": true` and `"workspaces": ["extension", "native-host", "raycast-extension"]` BEFORE running `changeset init`.
**Warning signs:** `changeset add` only offers the root package, not sub-packages.

### Pitfall 2: Private Package Releases Silently Fail
**What goes wrong:** `changesets/action` runs successfully but creates no GitHub Releases.
**Why it happens:** The action's `createGithubReleases` feature only works for packages that were actually published to npm. Private packages skip publishing, so no release is created.
**How to avoid:** Set `createGithubReleases: false` in the action config and use `softprops/action-gh-release@v2` separately, keyed off the `published` output.
**Warning signs:** Workflow succeeds but no release appears on the GitHub Releases page.

### Pitfall 3: GITHUB_TOKEN Tag Pushes Don't Trigger Workflows
**What goes wrong:** A separate `on: push: tags: ['native-host@*']` workflow never runs.
**Why it happens:** Events created by GITHUB_TOKEN (except workflow_dispatch/repository_dispatch) do not trigger additional workflows. This is by design to prevent infinite loops.
**How to avoid:** The release creation step MUST be in the same workflow run as `changeset publish`. Do not rely on tag-triggered workflows.
**Warning signs:** Tags appear in the repo but the release workflow shows no runs.

### Pitfall 4: macOS Runner Costs
**What goes wrong:** CI bills are higher than expected.
**Why it happens:** macOS runners cost 10x Linux runners per minute. Running all CI on macOS is expensive.
**How to avoid:** This project is macOS-only (Raycast requirement), so macOS is necessary. Keep workflows lean -- cache aggressively, avoid unnecessary steps.
**Warning signs:** Unexpected billing spikes. Monitor Actions usage in repo settings.

### Pitfall 5: npm ci Fails Without package-lock.json
**What goes wrong:** `npm ci` errors out because there is no lockfile.
**Why it happens:** The project currently has no root `package-lock.json`. Each sub-package may or may not have one. When adding workspaces, a single root lockfile is created.
**How to avoid:** After creating root `package.json` with workspaces, run `npm install` locally to generate the root `package-lock.json` and commit it.
**Warning signs:** CI fails on `npm ci` with "lockfile missing" error.

### Pitfall 6: Changesets Version PR Merge Conflicts
**What goes wrong:** The automated "Version Packages" PR conflicts with other PRs that modify `package.json` or `CHANGELOG.md`.
**Why it happens:** The version PR touches every changed package's `package.json` (version field) and `CHANGELOG.md`.
**How to avoid:** Merge the version PR promptly. The changesets action auto-updates the PR on each main push, resolving most conflicts.
**Warning signs:** Version PR shows merge conflicts in GitHub UI.

## Open Questions

1. **Exact GitHub repo name for changelog config**
   - What we know: `@changesets/changelog-github` needs `{ "repo": "owner/repo" }` in config
   - What's unclear: The exact GitHub username/org and repo name
   - Recommendation: Use the actual repo path; planner should reference it as a variable to fill in

2. **Raycast extension build in CI**
   - What we know: `ray build` and `ray lint` require the Raycast CLI
   - What's unclear: Whether `ray` CLI is available on GitHub Actions macOS runners or needs installation
   - Recommendation: Check if `@raycast/api` devDependencies provide the CLI, or if a separate install step is needed. The planner should include a verification step.

3. **web-ext availability for extension linting**
   - What we know: `extension/package.json` has `web-ext` as a devDependency
   - What's unclear: Whether `npm ci` at root workspace level installs workspace devDependencies
   - Recommendation: npm workspaces DOES install devDependencies of workspace packages by default. Should work with `npm run lint --workspace=extension`.

4. **Whether changesets `published` output fires for private packages**
   - What we know: `changeset publish` with `privatePackages.tag: true` creates git tags for private packages
   - What's unclear: Whether the `changesets/action` `published` output returns `true` when only private packages are tagged (no npm publishing)
   - Recommendation: The planner should include a fallback detection mechanism (e.g., checking if new tags were created via `git tag --points-at HEAD`) in case `published` is not set for private-only repos. This is the highest-risk uncertainty in the phase.
