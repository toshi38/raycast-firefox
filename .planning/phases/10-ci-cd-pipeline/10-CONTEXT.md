# Phase 10: CI/CD Pipeline - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Automate building and publishing the native host bundle via GitHub Actions. Every push to main with changesets automatically produces verified, downloadable native host artifacts as GitHub Releases. PR checks validate all components before merge.

</domain>

<decisions>
## Implementation Decisions

### Versioning & tagging
- Semantic versioning (semver) for all releases
- Changesets manages versions for all three components (extension, native-host, raycast-extension)
- Continuous delivery: push to main triggers changeset collection, version bumps, tagging, and release creation automatically
- Build and test must pass before release job runs
- Scoped tags per component: `native-host@1.0.0`, `extension@1.0.0`, `raycast-extension@1.0.0`

### Release format
- Release notes generated by changesets' built-in changelog (from changeset descriptions)
- Release assets: `host.bundle.js`, `run.sh`, `SHA256SUMS.txt` (generic names, no version in filename)
- The release/tag carries version info — asset filenames stay consistent across versions

### CI verification
- Full test suite runs against the built bundle before release is allowed
- SHA256 checksum generated and verified in CI before uploading (round-trip check)
- PR checks run build + full test suite on every pull request
- All three components (native-host, extension, raycast-extension) checked on PRs

### Workflow scope
- Two GitHub Actions workflows:
  1. **CI workflow** — triggers on PRs and main pushes: build, lint, test all components
  2. **Release workflow** — triggers on main after CI passes: changesets version, tag, publish GitHub Release with assets
- macOS ARM64 runner only (macOS-only tool, no cross-platform needed)
- npm dependency caching enabled for faster runs
- Uses built-in GITHUB_TOKEN (not a custom PAT — avoids infinite trigger loops)

### Claude's Discretion
- Exact GitHub Actions step configuration and action versions
- Job naming and workflow file structure
- How to handle changeset bot PR vs direct release
- Lint tooling choices for the CI check
- Test runner configuration for CI environment

</decisions>
