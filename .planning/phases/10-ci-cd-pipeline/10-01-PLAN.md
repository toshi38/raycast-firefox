  <action>
Create a root `package.json` with npm workspaces:

```json
{
  "name": "raycast-firefox",
  "version": "0.0.0",
  "private": true,
  "workspaces": [
    "extension",
    "native-host",
    "raycast-extension"
  ],
  "devDependencies": {
    "@changesets/cli": "^2.29",
    "@changesets/changelog-github": "^0.5"
  },
  "scripts": {
    "version": "changeset version",
    "release": "changeset publish"
  }
}
```

Create `.changeset/config.json`:

```json
{
  "$schema": "https://unpkg.com/@changesets/config@3.1.1/schema.json",
  "changelog": ["@changesets/changelog-github", { "repo": "toshi38/raycast-firefox" }],
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": [],
  "privatePackages": {
    "version": true,
    "tag": true
  }
}
```

CRITICAL: `privatePackages.tag: true` is required so `changeset publish` creates git tags like `native-host@1.0.0`. Without this, no tags are created and releases never happen.

Also create `.changeset/README.md` (standard changesets convention -- a brief one-liner explaining what the directory is for).

Update `.gitignore` to add root-level entries:
- `node_modules/` (root workspace node_modules)
- Keep existing entries intact

Then run `npm install` at the root to generate `package-lock.json` and hoist workspace dependencies. This will create a root `node_modules/` with symlinks to each workspace. Commit the root `package-lock.json` (CI needs it for `npm ci`).

Delete the old per-workspace lockfiles (`extension/package-lock.json`, `native-host/package-lock.json`, `raycast-extension/package-lock.json`) since the root lockfile now governs all workspace dependencies. These old files will cause confusion and potential conflicts with the workspace root lockfile.

Verify: `npm ls --workspaces` shows all three packages. `npx changeset status` runs without error.

<task type="auto">
  <name>Task 2: Create CI workflow for PRs and main pushes</name>
  <files>
    .github/workflows/ci.yml
  </files>
  <action>
Create `.github/workflows/ci.yml` with the following configuration:

**Triggers:** `pull_request` targeting `main` branch ONLY. Do NOT include `push: branches: [main]` — the release workflow (10-02) handles main-push CI as a prerequisite job with `needs: [ci]`, ensuring builds/lints must pass before release runs. Having CI also trigger independently on main push would be redundant and would NOT gate the release.

**Runner:** `macos-latest` (ARM64 by default as of 2025).

**Steps:**
1. `actions/checkout@v4`
2. `actions/setup-node@v4` with `node-version: 20` and `cache: npm`
3. `npm ci` (installs all workspace dependencies from root lockfile)
4. **Native host build:** `npm run build --workspace=native-host`
5. **Native host bundle verification:** `test -f native-host/dist/host.bundle.js` (ensures the build actually produced output)
6. **Firefox extension lint:** `npm run lint --workspace=extension` (uses web-ext lint)
7. **Raycast extension lint:** `npm run lint --workspace=raycast-extension` (uses ray lint)
8. **Raycast extension build:** `npm run build --workspace=raycast-extension` (uses ray build)

Name the job `ci`. Name the workflow `CI`.

Do NOT include test steps beyond build + lint -- the project does not have a test suite yet. Do NOT include the Raycast extension build if `ray` CLI is not available on GitHub Actions runners. Instead, add a comment noting this step may need adjustment once CI runner availability of `ray` is confirmed.

NOTE on Raycast CLI availability: The `ray` CLI is installed via the Raycast app, not via npm. It may NOT be available on GitHub Actions macOS runners. For the Raycast extension steps, wrap them with `if: false` and add a comment: `# TODO: Enable when ray CLI availability on CI is confirmed`. The lint and build for the extension workspace rely on `ray lint` and `ray build` which require the Raycast desktop app CLI. For now, only the Firefox extension lint and native-host build are guaranteed to work.
