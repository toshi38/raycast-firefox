  <action>
Create `.github/workflows/release.yml` with TWO jobs: `ci` and `release`. The `release` job has `needs: [ci]` so build and test must pass before release runs. This satisfies the locked decision: "Build and test must pass before release job runs."

**Trigger:** `push` to `main` branch only.

**Concurrency:** `${{ github.workflow }}-${{ github.ref }}` to prevent duplicate release runs.

**Permissions (workflow-level):** `contents: write`, `pull-requests: write` (needed for creating releases and version PRs).

**Runner:** `macos-latest` (for both jobs)

---

**Job 1: `ci`** (build and lint gate)

Replicates the same CI checks as `ci.yml` (which only runs on PRs). This ensures main pushes also get validated before release.

Steps:
1. `actions/checkout@v4`
2. `actions/setup-node@v4` with `node-version: 20` and `cache: npm`
3. `npm ci`
4. **Native host build:** `npm run build --workspace=native-host`
5. **Native host bundle verification:** `test -f native-host/dist/host.bundle.js`
6. **Firefox extension lint:** `npm run lint --workspace=extension`
7. Raycast extension steps with `if: false` (same as ci.yml — pending ray CLI availability)

---

**Job 2: `release`** (`needs: [ci]`)

Steps:
1. `actions/checkout@v4` with `fetch-depth: 0` (needed for tag detection fallback)
2. `actions/setup-node@v4` with `node-version: 20` and `cache: npm`
3. `npm ci`
4. Build native host: `npm run build --workspace=native-host`
5. **Changesets action:**
   ```yaml
   - name: Create Release Pull Request or Publish
     id: changesets
     uses: changesets/action@v1
     with:
       publish: npx changeset publish
       version: npx changeset version
       commit: "chore: version packages"
       title: "chore: version packages"
       createGithubReleases: false
     env:
       GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
   ```
   CRITICAL: `createGithubReleases: false` because the built-in release creation does NOT work for private packages. We handle it ourselves below.

6. **Fallback tag detection** (runs always, derives `should_release`):
   ```yaml
   - name: Detect release condition
     id: detect-release
     run: |
       # Primary: trust changesets published output
       if [ "${{ steps.changesets.outputs.published }}" = "true" ]; then
         echo "should_release=true" >> "$GITHUB_OUTPUT"
         echo "Release detected via changesets published output"
       else
         # Fallback: check if changesets just created a native-host tag on HEAD
         NH_TAG=$(git tag --points-at HEAD | grep '^native-host@' || true)
         if [ -n "$NH_TAG" ]; then
           echo "should_release=true" >> "$GITHUB_OUTPUT"
           echo "Release detected via fallback tag detection: $NH_TAG"
         else
           echo "should_release=false" >> "$GITHUB_OUTPUT"
           echo "No release detected"
         fi
       fi
   ```
   This addresses the uncertainty about whether `changesets/action` sets `published: true` for private-package-only repos. If `published` never fires, the fallback detects tags created by `changeset publish`.

7. **Generate SHA256 checksum** (conditional on `steps.detect-release.outputs.should_release == 'true'`):
   ```bash
   cd native-host/dist
   shasum -a 256 host.bundle.js run.sh > SHA256SUMS.txt
   shasum -a 256 -c SHA256SUMS.txt
   ```
   Uses `shasum -a 256` (macOS built-in, not `sha256sum` which is Linux). The round-trip verify (`-c`) confirms the checksums match what was just generated.

8. **Get native-host version** (conditional on should_release):
   ```bash
   VERSION=$(node -p "require('./native-host/package.json').version")
   echo "version=$VERSION" >> "$GITHUB_OUTPUT"
   echo "tag=native-host@$VERSION" >> "$GITHUB_OUTPUT"
   ```

9. **Create GitHub Release** (conditional on should_release):
   ```yaml
   - name: Create GitHub Release
     if: steps.detect-release.outputs.should_release == 'true'
     uses: softprops/action-gh-release@v2
     with:
       tag_name: ${{ steps.nh-version.outputs.tag }}
       name: native-host@${{ steps.nh-version.outputs.version }}
       generate_release_notes: false
       files: |
         native-host/dist/host.bundle.js
         native-host/dist/run.sh
         native-host/dist/SHA256SUMS.txt
   ```

10. **Post-release verification** (conditional on should_release):
    ```yaml
    - name: Verify GitHub Release was created
      if: steps.detect-release.outputs.should_release == 'true'
      run: |
        TAG="${{ steps.nh-version.outputs.tag }}"
        # Verify the release exists via gh CLI
        gh release view "$TAG" --json tagName,assets --jq '.assets[].name' | sort
        echo "Release $TAG verified with assets above"
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    ```
    This ensures the workflow fails visibly if the release was supposed to be created but wasn't.

**IMPORTANT ANTI-PATTERNS TO AVOID:**
- Do NOT use a PAT instead of GITHUB_TOKEN (causes infinite workflow triggers)
- Do NOT create a separate tag-triggered workflow (GITHUB_TOKEN pushes don't trigger workflows)
- Do NOT set `createGithubReleases: true` (silently produces no releases for private packages)
- Do NOT use `npm install` instead of `npm ci`
- Do NOT omit `fetch-depth: 0` on the release job checkout (needed for `git tag --points-at HEAD` fallback)

Name the workflow `Release`.
